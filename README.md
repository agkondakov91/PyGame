# 🎮 Pygame Runner — учебный курс

Учебный курс по разработке 2D runner-игры на **Pygame**.

В рамках курса студенты постепенно создают небольшую игру: персонаж бежит по уровню, прыгает, собирает монеты, избегает препятствий, набирает очки, проигрывает при столкновении и может начать новую попытку.

Курс показывает не только базовые возможности Pygame, но и постепенное развитие проекта: от одного простого файла до более аккуратной структуры с модулями, классами, менеджерами объектов, интерфейсом и отдельным состоянием игровой сессии.

## 🚀 Быстрый старт

### 1. Клонировать репозиторий

```bash
git clone git@github.com:agkondakov91/PyGame.git
cd PyGame
```

### 2. Проверить uv

```bash
uv --version
```

Если `uv` не установлен:

```bash
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

# Windows PowerShell
powershell -c "irm https://astral.sh/uv/install.ps1 | iex"
```

### 3. Установить Python 3.13

```bash
uv python install 3.13
```

В проекте версия Python закреплена в файле `.python-version`:

```text
3.13
```

### 4. Установить зависимости

```bash
uv sync --dev
```

Эта команда установит основные зависимости проекта и dev-инструменты.

### 5. Запустить игру

```bash
uv run python main.py
```

## 🧩 Архитектура проекта

Проект постепенно развивается от простого игрового цикла к более аккуратной архитектуре.

### `main.py`

Точка входа в приложение.

```python
from src.runner.game import Game


def main() -> None:
    game = Game()
    game.run()


if __name__ == "__main__":
    main()
```

`main.py` не содержит игровой логики. Он только создаёт объект игры и запускает главный цикл.

---

### `Game`

Главный координатор игры.

Класс `Game` отвечает за:

* инициализацию Pygame;
* создание окна;
* главный игровой цикл;
* обработку событий;
* вызов обновления объектов;
* проверку столкновений;
* вызов отрисовки;
* завершение игры.

`Game` не должен вручную управлять каждой монетой, каждым препятствием, каждым текстом интерфейса и каждой настройкой состояния. Для этого есть отдельные классы.

---

### `GameSession`

Класс `GameSession` хранит данные текущей игровой сессии:

* текущее состояние игры;
* текущий счёт;
* лучший результат;
* множитель скорости.

Он отвечает за действия:

```python
self.session.start_game()
self.session.pause()
self.session.resume()
self.session.finish_game()
self.session.go_to_menu()
self.session.add_score(points)
self.session.increase_speed()
```

Это делает код выразительнее: вместо прямого изменения строк и чисел используются методы, отражающие смысл игровых действий.

---

### `UIManager`

Класс `UIManager` отвечает за отрисовку интерфейса:

* стартовое меню;
* текущий счёт;
* лучший результат;
* множитель скорости;
* экран паузы;
* экран Game Over.

Благодаря этому `Game` не хранит шрифты и не содержит координаты каждого текста.

---

### `BaseObjectManager`

Базовый менеджер игровых объектов.

Он содержит общее поведение для групп объектов:

* создать несколько объектов;
* сбросить их позиции;
* обновить все объекты;
* пересоздать объект, если он вышел за экран;
* нарисовать все объекты.

От него наследуются:

```text
CoinManager
ObstacleManager
```

---

### `CoinManager`

Менеджер монет.

Он управляет списком монет, обновляет их, рисует и проверяет сбор игроком.

При сборе монеты счёт увеличивается, а монета появляется снова дальше по уровню.

---

### `ObstacleManager`

Менеджер препятствий.

Он управляет списком препятствий, обновляет их, рисует и проверяет столкновение игрока с препятствием.

При столкновении игра переходит в состояние Game Over.

---

### `SpawnManager`

Класс, который отвечает за появление объектов справа от экрана.

Он выдаёт следующую координату по оси `x` с учётом случайного промежутка.

Для препятствий и монет используются разные диапазоны:

```python
OBSTACLE_SPAWN_MIN_GAP = 450
OBSTACLE_SPAWN_MAX_GAP = 750

COIN_SPAWN_MIN_GAP = 250
COIN_SPAWN_MAX_GAP = 500
```

Это позволяет делать препятствия более редкими, а монеты — более частыми.

## 🔁 Игровой цикл

Основа Pygame-игры — игровой цикл:

```python
while self.running:
    self.handle_events()

    if self.session.is_playing():
        self.update()

    self.draw()

    pygame.display.update()
    self.clock.tick(FPS)
```

Каждый кадр игра:

1. обрабатывает события;
2. обновляет игровой мир;
3. рисует новый кадр;
4. обновляет экран;
5. ограничивает FPS.

Если игра находится в меню, на паузе или на экране Game Over, игровой мир не обновляется.

## 🧠 Состояния игры

В проекте используются состояния:

```python
STATE_MENU = "menu"
STATE_PLAYING = "playing"
STATE_PAUSE = "pause"
STATE_GAME_OVER = "game_over"
```

Схема переходов:

```text
menu
  ↓ Enter / Space

playing
  ↓ P
pause
  ↓ P
playing

playing
  ↓ столкновение с препятствием
game_over

game_over
  ↓ R
playing

game_over
  ↓ Esc
menu
```

Состояния помогают разделить поведение игры. В меню мы показываем стартовый экран, во время игры обновляем мир, на паузе замораживаем обновление, а после проигрыша показываем результат.

## 💥 Столкновения и хитбоксы

Pygame позволяет проверять столкновения через `Rect`:

```python
player_hitbox.colliderect(obstacle_hitbox)
```

Но изображения часто содержат прозрачные области. Поэтому для более честных столкновений используются хитбоксы:

```python
PLAYER_HITBOX_INFLATE = (-40, -30)
OBSTACLE_HITBOX_INFLATE = (-30, -40)
COIN_HITBOX_INFLATE = (-15, -15)
```

Хитбокс меньше исходного прямоугольника изображения, поэтому столкновения ощущаются точнее.

## 🐞 Частые проблемы

### `ModuleNotFoundError: No module named 'pygame'`

Скорее всего, проект запускается не через `uv`.

Правильно:

```bash
uv run python main.py
```

Неправильно:

```bash
python main.py
```

---

### Картинка не загружается

Проверь:

* файл лежит в `assets/images/`;
* имя файла совпадает с тем, что указано в коде;
* расширение файла совпадает: `.png`, `.jpg`, `.jpeg`;
* проект запускается из корня репозитория.

---

### Звук не воспроизводится

Проверь:

* файл лежит в `assets/sounds/`;
* имя файла совпадает с кодом;
* окно Pygame активно;
* звук не выключен в системе;
* формат файла поддерживается.

---

### Столкновение происходит слишком рано

Проблема обычно в том, что исходный `Rect` изображения больше видимой части объекта.

Решение — использовать уменьшенный хитбокс:

```python
player_hitbox = player.rect.inflate(*PLAYER_HITBOX_INFLATE)
```

---

### Монеты или препятствия появляются слишком редко

Проверь значения в `constants.py`:

```python
OBSTACLE_SPAWN_MIN_GAP = 450
OBSTACLE_SPAWN_MAX_GAP = 750

COIN_SPAWN_MIN_GAP = 250
COIN_SPAWN_MAX_GAP = 500
```

Если объекты появляются слишком редко, диапазоны можно уменьшить.

Если появляются слишком часто, диапазоны можно увеличить.
